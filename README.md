Все обязательные поля должны быть заполнены при создании записи.

Данные должны вводиться в соответствии с утверждёнными справочниками (должности, подразделения, категории, локации, коды).

Даты должны быть логически корректны: дата увольнения не может быть раньше даты приема, дата документа не может быть в будущем, дата рождения должна соответствовать нормам.

Числовые значения должны быть реалистичными и попадать в допустимые диапазоны: оклад больше нуля, FTE от 0 до 1, KPI от 0 до 100 и т. д.

Запрещается создавать дубли: сотрудник должен иметь уникальный табельный номер, вакансия — уникальный идентификатор, клиент — уникальный контакт.

Данные должны быть внесены своевременно, не позднее установленного регламентом срока (например, увольнение — в день подписания приказа).

Все записи должны соответствовать бизнес-правилам: корректные статусы, согласованность подразделения и должности, правильная принадлежность к структуре.

Первичные данные не должны редактироваться вручную в обход стандартных процессов; все изменения проходят только через предусмотренные процедуры.

Данные должны быть единообразными во всех системах: одинаковые названия, единые форматы дат, одинаковые значения справочников.

Первичные записи должны быть проверяемыми: должна сохраняться история внесения изменений и автор записи.

Не допускаются текстовые значения вместо структурированных данных: нельзя вводить должность вручную в поле комментариев, нельзя вносить суммы текстом.

Значения должны иметь правильный формат: даты в формате YYYY-MM-DD, суммы — в числовом формате, идентификаторы — без лишних символов.

Пользователь, создающий данные, должен иметь соответствующие полномочия, а поля должны вводиться только через утвержденные формы и механизмы.

Первичные данные должны быть однозначными и не допускать двойной трактовки (например, статус «в работе» должен означать одно и то же во всех процессах).

Должны использоваться только актуальные версии справочников, моделей и классификаторов.
